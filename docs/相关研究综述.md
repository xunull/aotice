## 1. Expensively Quadratic: the LLM Agent Cost Curve(exe.dev)

**定位**:离我们核心洞察最近的一篇。作者把 coding agent 的成本曲线拆开看。

**核心论点**:
- Agent 在循环里反复把整段对话历史发给模型,每次调用都要为**已缓存的历史按缓存读价重新计费**。
- 关键金句:**"缓存读成本不是 token 数的平方,而是 token 数 × 调用次数"**——因为调用次数本身随对话增长,所以合起来是二次的。
- 在他的一个真实会话里(总花费 $12.93),**缓存读占了最终成本的 87%**,并在仅 **27,500 token** 处就与"下一次调用的其它成本"打平。用 Opus 4.5 定价(写 1.25×、输出 5×、读 0.1×)模拟,**大约 20,000 token 缓存读就开始主导**。

**建议的对策**:① 减少调用次数;② 用子 agent 把迭代工作丢到主上下文之外;③ **干脆重开会话**(重置上下文成本,常比续着更便宜);④ 别把工具输出截断成多次请求。

**与我们的关系**:**直接印证**我们文档第 1 节的前提——读税(缓存读)随上下文二次增长、是长会话真正烧钱的地方。区别:他停在"识别问题 + 给经验对策",**没有把"何时压缩"建成一个可求解的最优问题**。我们从这里往前走了一步。

🔗 https://blog.exe.dev/expensively-quadratic

## 2. Don't Break the Cache（arXiv 2601.06007）

**定位**:专门研究"上下文管理动作 vs 缓存"的矛盾,正好对应我们模型里的**缓存重建成本**。

**核心论点**:
- Prompt caching 在多轮 agent 里能显著降本降延迟,但**效果高度依赖缓存的稳定性**。
- 中心张力:当 agent 在步骤之间**动态修改、压缩、摘要**上下文时,**缓存命中率大幅下跌**——常见的"渐进更新上下文、过滤信息、压缩历史"等做法会无意中让缓存失效,把收益吃掉。

**建议**:保持**"上下文稳定性"**——保留原始上下文而非改写已缓存段;把**可变内容和缓存块分离**;把频繁变化的元素隔离到末尾;精心设计 agent loop 以减少打破缓存的编辑。

**与我们的关系**:**印证**我们模型里"压缩会打破缓存、下一轮要重写摘要"这一项固定开销(`K_fix` 里的写价部分),也支撑我们的**建议 #3**:比一刀切摘要更好的是结构化 context editing(只清除陈旧 `tool_result`、保留前缀稳定)。这篇论文相当于给"别为了压缩而牺牲缓存"提供了学术证据。

🔗 https://arxiv.org/pdf/2601.06007

## 3. The Complexity Trap：观察遮蔽 vs LLM 摘要（arXiv 2508.21433）

**定位**:从**质量维度**回答"该怎么管理上下文",而不只是成本。

**研究问题**:软件工程 agent 真的需要复杂的 LLM 摘要吗?还是简单的"观察遮蔽"(把超过固定窗口 M=10 轮的工具输出用占位符遮掉、但保留推理与动作)就够了?

**关键实验发现**(SWE-bench Verified,500 例,5 种模型配置):
- **两种策略都比不管理省 >50% 成本。**
- **观察遮蔽在 5 组里有 4 组成本最低**;解题率还能打平甚至略高(如 54.8% vs 摘要的 53.8%)。
- **LLM 摘要让轨迹变长 13–15%**,部分抵消了它的节省;摘要调用本身最多吃掉 **7.2%** 的成本。
- 最佳是**混合策略**(遮蔽 + 摘要),额外再省 **7–11%** 且解题率更高。

**结论/建议**:把观察遮蔽作为**默认**策略;LLM 摘要留给**极长轨迹(>31 轮)当最后手段**。作者明确**质疑业界一窝蜂上 LLM 摘要**的潮流。

**与我们的关系**:**强力支撑**我们的建议——"比一刀切摘要更优的是结构化裁剪(clear_tool_uses)"。它还提醒:我们那个"压缩固定开销"其实偏低估了 LLM 摘要的真实代价(摘要会让轨迹变长→更多轮→更多读税),这正是我们模型没货币化、但会把最优点右移的因素之一。

🔗 https://arxiv.org/html/2508.21433

## 4. Context Engineering for Agents（TrueFoundry）

**定位**:工程实践指南,给出**具体阈值经验值**,可与我们的闭式解对照。

**核心建议与数据**:
- **在窗口的 75% 处主动压缩**(200K 窗口 → 150K 触发)。强调**主动**,别等撞上限,否则最后几轮质量会掉。
- **别压太早**:**低于 ~75K token 或第 10 轮之前不要压**,开销大于收益。
- **缓存是最高 ROI**:缓存读按输入价的 10% 计费(Sonnet 4.6 是 $0.30/M);一个 12K 系统提示每小时用 1000 次,靠 85% 命中率从 $36/时降到约 $9.81/时(**降 73%**)。稳定内容在前、可变内容在后,保持前缀逐字节一致。
- **真实 agent 每轮约积累 6K token**,从 12K(系统提示)一路爬到 30 轮后的 200K+。
- 优先用**服务端压缩**(`context_management`),省一次往返。

**与我们的关系**:**最直接的对照**。它代表"业界拍经验值"这一派——75% 是个合理的保守数,但**不是从成本模型推出来的**。我们的贡献正是把这个数变成可算的(并解释为什么纯成本最优往往远早于 75%,是质量因素把它推回去)。另外它的 **"每轮 ~6K token"** 佐证了我们计算器里 `g` 的默认量级。

🔗 https://www.truefoundry.com/blog/context-engineering-gateway-session-management

## 5. How We Cut LLM Costs by 59% With Prompt Caching（ProjectDiscovery）

**定位**:生产环境实战复盘(Neo 安全测试 agent,每任务 20–40+ 步),给出**缓存失效的真实量级**。

**关键技术**:
- **三个缓存断点**:BP1 静态系统提示(1h TTL,跨用户共享)、BP3 工具定义(静态在前动态在后)、BP2 会话滑动窗口(5min TTL 打在最后一个工具结果上)。
- **"搬家"技巧(收益最大)**:把动态内容(工作记忆、skills、运行时上下文)从系统消息**搬到末尾的 `<system-reminder>` 用户消息**,避免前缀中段变化击穿下游缓存——**命中率一夜从 7% 跳到 74%**。
- 配套:模板变量用占位符、时间戳冻结到"日"粒度、路由到 Anthropic Direct 保持缓存局部性。

**成本数据**:总体**降 59%**;优化后 66%;近 10 天 70%;**命中率从 7% → 84%**;累计 98 亿 token 从缓存读取;极端任务 6750 万输入 token、91.8% 命中率。

**实战教训**:① 缓存是多步 agent 最高 ROI 的基础设施;② **动态内容的位置至关重要,中段变化会静默杀掉命中**;③ 跨用户共享 + 长 TTL 放大收益;④ 按流(per-stream)而非聚合来度量。

**与我们的关系**:**实证**了两件事——(a)缓存读/击穿的成本量级是真实且巨大的(命中率 7%→84% 差一个数量级),支撑我们把"缓存重建"当成一项真成本;(b)"前缀中段一变就击穿"正是我们文档"缓存 TTL/失效"一节和 Don't Break the Cache 讲的同一现象的生产级注脚。

🔗 https://projectdiscovery.io/blog/how-we-cut-llm-cost-with-prompt-caching

## 综合:这些工作共同勾勒出什么

把五篇拼起来,能看到一条清晰的共识链:

1. **问题已被公认**:上下文二次成本、缓存读主导(#1、#5)。
2. **缓存是主要杠杆,但脆弱**:压缩/改写会击穿缓存,收益被吃掉(#2、#5)。
3. **管理策略之争偏向"简单裁剪 > 复杂摘要"**:至少在成本和质量的综合上(#3)。
4. **触发时机仍是经验值**:75%、50%、第 10 轮之后……都是拍出来的启发式(#4)。

**我们相对他们新在哪**:没有一篇**把"何时触发压缩"显式建成成本最小化问题并给出闭式解**。我们用 EOQ(经济订货量)把#1 的读税当持有成本、把#2/#5 的缓存重建 + 摘要生成当每次固定开销,推出 `T* = S + √(2·g·K/h)`,并接上真实定价做成可交互工具。

**我们相对他们的局限**:#3 提醒我们低估了 LLM 摘要的隐性成本(轨迹变长),#4 提醒真实决策由质量主导——所以我们那个"纯成本最优 ~13%"只是**下界参照**,不是要你真压到那么早。这也是为什么这份综述值得读:它把我们模型的**边界和适用范围**钉清楚了。
