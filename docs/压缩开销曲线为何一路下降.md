# 压缩开销曲线为何一路下降(三条曲线的形状)

> 计算器那张图里有三条线:U 形的**总成本**、往上走的**读税**、往下走的**压缩开销**(橙色)。很多人第一次看会卡在橙线上:它随阈值一路下降、趋近于零——那岂不是"压得越晚越好、永远别压最省"?这篇把这条线的形状讲透,并说明为什么它必须配着那条往上走的读税一起看,谷底才是最优。
>
> 关联:[成本建模总览](./上下文压缩时机的成本建模.md) · [公式详解](./重读放大与公式详解.md) · [EOQ 详解](./EOQ经济订货量详解.md)

---

## 1. 先对齐:横轴、橙线各是什么

- **横轴 T** = 压缩触发阈值。上下文涨到 T 就压一次(占窗口的百分比或绝对 token 数)。
- **橙线 = 压缩开销**,公式在图下方:

```
压缩开销 = g/(T − S) · K_fix
```

| 符号 | 含义 | 默认值 |
|---|---|---|
| `g` | 每轮新增 token(需求率) | 5000 |
| `S` | 压缩后地板(摘要 + 近期) | 20000 |
| `T` | 触发阈值(横轴) | 变量 |
| `K_fix` | **每压一次的固定开销** = `amp·S·(输出价 + 写价)` | ≈ 0.625 |

`K_fix` 是"压一次要花多少钱":写摘要要输出 token(按输出价),重建的上下文要写进缓存(按缓存写价),`amp` 是压缩后重读文件的放大。**它跟你在哪个阈值压关系不大,近似是一笔固定费用。**

---

## 2. 橙线为什么一路下降:它是"摊到每轮"的压缩成本

关键一步:把 `压缩开销 = g/(T−S) · K_fix` 读成 **频率 × 单价**。

```
压缩开销  =   g/(T − S)      ×      K_fix
            └────┬────┘            └──┬──┘
             压缩频率              每次固定开销
           (次 / 每轮)             (固定)
```

- 从地板 `S` 涨到阈值 `T`,你才压一次。所以**每次压缩"买到"的运行空间 = `T − S`**。
- 每轮涨 `g`,于是压缩频率 ≈ `g / (T − S)` 次每轮。
- 频率乘单价 `K_fix`,就是**摊到每一轮的压缩成本**。

**T 越大 → 每次压缩买到的空间 `T−S` 越大 → 同样长的对话里压缩次数越少 → 摊到每轮越薄。** 分母 `T−S` 随 T 变大,整条线就随 T 单调下降,趋近于零。

> 一句话类比:压缩就是"下单进货"。每次下单有固定手续费(`K_fix`)。**一次进得多(阈值 T 高)→ 下单次数少 → 摊下来的手续费低。** 这就是橙线往下走的全部原因。

## 3. 拿默认参数算给你看(每 100 轮)

`g=5000, S=20000, K_fix≈0.625`,纵轴按图里的"每 100 轮"口径:

| 阈值 T | 压缩节奏 | **压缩开销(橙)** | 读税(青) | 总成本 |
|---|---|---|---|---|
| 50K | 频繁压 | **$10.4** | $1.8 | $12.2 |
| 132K(最优 T*) | 中等 | **$2.8** | $3.8 | **$6.6** ← 谷底 |
| 500K | 很少压 | **$0.65** | $13.0 | $13.7 |

橙线从 $10.4 一路掉到 $0.65,确实单调下降。但看最后一列:总成本并不是跟着橙线一起降——它先降后升,谷底在 132K。

---

## 4. 为什么不是"越高越好":橙线有个死对头

如果只盯橙线,结论会是"永远别压"(`T→∞` 时压缩开销 → 0)。错在**只看了一半**。

另一半是**青色的读税(持有税)**,它是**上升**的:

```
读税 = h · (S + T) / 2        h = 缓存读价
```

上下文留得越大(T 越高),你**每一轮都要按缓存读价重读整段上下文**,这笔持有税就越高。看上表:T 从 50K 涨到 500K,读税从 $1.8 涨到 $13.0。

两条线方向相反:

```
成本
 │＼                              ╱  总成本(U 形)= 橙 + 青
 │  ＼                        ╱
 │    ＼      总成本谷底    ╱
 │      ＼      ↓ T*      ╱
 │  橙    ＼    ┊       ╱    青
 │(压缩开销)＼  ┊     ╱  (读税)
 │  压得少省    ＼┊  ╱    上下文大就贵
 │  ↘↘↘↘↘↘      ╳       ↗↗↗↗↗↗
 │            ╱ ┊ ＼
 └──────────┴──┴──┴────────────────→  阈值 T
                T*(虚线)
```

- **橙线(压缩开销)**:压得越少越省 → 往下
- **青线(读税)**:上下文越大越贵 → 往上
- **两条一加 = U 形总成本**,谷底那条虚线就是**最优阈值 `T*`**

这正是 [EOQ(经济订货量)](./EOQ经济订货量详解.md)的经典权衡:**批量下单省手续费,但囤货涨持有费**,中间有个最优批量。压缩 = 下单,上下文大小 = 库存。`T* = S + √(2·g·K_fix/h)` 就是这个 U 形谷底的闭式解。

---

## 5. 常见误读与结论

**误读**:"橙线一路降,所以阈值越高越省 / 干脆别压。"
**为什么错**:橙线只是权衡的一半。它降是因为"压得少、固定开销摊得薄";但你为此囤了更大的上下文,青色读税会加倍还回来,而且读税随上下文**线性**涨、没有上限。

**正确读法**:这三条线要**成对看**。

- 单看橙线 → 会把最优点推到无穷远(永不压缩)。
- 单看青线 → 会把最优点推到最小(每轮都压)。
- 两条相加 → U 形,谷底 `T*` 才是真正省钱的点。

**一句话**:橙线一路下降,说的是"压得越少,每次压缩的固定开销摊得越薄"这半个道理;它天生要跟"上下文越大、每轮重读越贵"的青线抵消。谁也不能单独定最优——`T*` 是两者的平衡点。

> 想直观感受:在计算器里把**重读放大系数 amp** 调大。`amp` 进 `K_fix`,会把橙线整体抬高(每次压缩更贵),谷底 `T*` 随之**右移**——因为压缩变贵了,就该少压、让上下文留得更大一些。参见[重读放大与公式详解](./重读放大与公式详解.md)。
