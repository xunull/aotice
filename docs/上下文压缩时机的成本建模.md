# 上下文压缩（compaction）时机的成本建模

> 讨论对象：Claude Code / Codex 这类 agentic CLI 在上下文接近窗口上限时触发的"压缩/摘要"（compaction）。
> 核心问题：**在上下文使用到多少比例时触发压缩，才是成本最优的？**
> 结论先行：**"晚点压缩 = 压缩次数少 = 更省钱" 在纯 token 成本上是错的**——真正的主导成本是"每轮的缓存读税"，它偏向**更早**压缩；而质量损失与重读放大又把最优点**往后**推。二者折中才是现实中的最优。

---

## 0. 记号与定价（基准）

| 符号 | 含义 | 取值示例（Opus 4.8） |
|---|---|---|
| `P` | 输入基础单价 | $5 / M token |
| `P_out` | 输出单价 | $25 / M token |
| `h = 0.1·P` | **缓存读**单价（持有成本，每 token 每轮） | $0.5 / M |
| `w = 1.25·P` | **缓存写**单价（5min TTL） | $6.25 / M |
| `w₁ₕ = 2·P` | 缓存写单价（1h TTL） | $10 / M |
| `W` | 上下文窗口上限 | 1,000,000 |
| `C` | 当前上下文大小 | — |
| `g` | 每轮新增 token（"需求率"） | 2,000–10,000 |
| `S` | 压缩后地板大小（摘要 + 保留的近期消息，≈固定） | ~20,000 |
| `T` | 压缩触发阈值（决策变量） | — |
| `N` | 会话总轮数 | — |

参考定价：Sonnet 5 为 P=$3/M、P_out=$15/M；缓存倍率相同（读 0.1×、写 5min 1.25× / 1h 2×）。

---

## 1. 被忽略的主导成本：每轮的"缓存读税"

agentic loop 里，**每一轮（每次 API 调用）都要把整段已累积上下文重新发一遍**。启用 prompt caching 后：

- 命中缓存的前缀（≈整段历史）按 **0.1×P**（cache read）计费；
- 本轮新增的增量 token 按 **1.25×P**（5min 写）计费；
- 输出单独按 `P_out`。

因此：**上下文有多大，你每一轮都按 `0.1·P·C` 把它整个"读"一遍。** 这是一笔

- 随上下文**线性增长**、
- 按轮数**不断重复**

的"持有税"。由于 C 随轮数线性增长，**整段会话的累计成本是二次增长的**。

> 这正是"用总量/单次容量去数压缩次数"这种直觉的盲区：它只数了压缩事件，没算这笔每轮都在滴血的读税。

---

## 2. 三项成本模型

压缩把上下文从阈值 `T` 砍回地板 `S`。每轮新增 `g`，会话共 `N` 轮。与阈值 `T` 相关的成本分三项：

1. **读税（持有成本）**：平均上下文 ≈ `(S+T)/2`，累计
   `≈ N · h · (S+T)/2` → **随 T 增大**（晚压缩 = 上下文更大更久 = 更贵）。
2. **压缩固定开销**：每次压缩 ≈ 生成摘要 `S·P_out` + 摘要首次写缓存 `S·w` ≡ **`K_fixed`**；
   压缩次数 `m ≈ N·g / (T−S)` → **随 T 增大而减少**（晚压缩 = 次数少）。
3. **压缩时读一遍全上下文**（`0.1·P·T`）：总量**近似**与 T 无关。严格展开 `T/(T−S) = 1 + S/(T−S)`：前一部分是真常数，后一部分等价于给 `K_fixed` 增加 `0.1·P·S`（默认参数下约 +1.6%，对 `T*` 的影响 <1%）。按常数项处理，**不影响最优点**。

> ⚠️ 前提：**摘要是固定大小 S**（而非"按比例压缩"）时，第 2 项才随 T 减少，从而出现内部最优。
> Claude Code / Codex 都是压到**有界摘要**（Codex 硬性保留最近 ~20K token + 定长 handoff 摘要），所以现实里确实有最优点。
> 若是纯按比例压缩，模型退化为"越早压越省钱"（无内部最优）。

---

## 3. 最优阈值 = 经济订货量（EOQ）平方根公式

把"每周期增长量" `ΔC = T − S` 当作批量，这就是教科书的 **EOQ 问题**：
持有成本 `h` ↔ 库存持有，`K_fixed` ↔ 每次订货固定成本，`g` ↔ 需求率。最优解：

```
             ┌────────────────────────┐          ┌──────────────────────────────┐
ΔC* = sqrt(  │  2 · g · K_fixed / h    │ ) = sqrt( │ 2·g·S·(P_out + 1.25P) / (0.1P) │ )
             └────────────────────────┘          └──────────────────────────────┘

T* = S + ΔC*
```

### 代入 Opus 4.8

- `h = 0.1·P = $0.5/M`
- `K_fixed = S·(P_out + 1.25P) = S·(25 + 6.25)/M = S·$31.25/M`；取 `S=20K` → `K_fixed ≈ $0.625/次`
- 取 `g = 5,000 token/轮`：

```
ΔC* = sqrt( 2 · 5000 · 0.625 / 0.5e-6 ) ≈ 112,000 token
T*  = 20K + 112K ≈ 132,000 token   （≈ 1M 窗口的 13%）
```

**即最省钱的触发点约在 130K，只占 1M 窗口的 ~13%——远早于 90%。**

### 直觉对照（Opus 4.8，仅读税）

| 稳态上下文 | 每轮读税 | 300 轮会话仅读税 |
|---|---|---|
| ~130K | $0.065 | ~$20 |
| ~500K | $0.25 | ~$75 |
| ~900K | $0.45 | ~$135 |

每次压缩固定开销才 ~$0.6。**只要不是压得极其频繁，读税就完全主导**——所以"早压缩"在纯成本上赢。

> ⚠️ 这张表是**上下文恒定在该值**的示意（便于建立直觉）。配套计算器里 `读税 = h·(S+T)/2` 建模的是**从地板 S 循环增长到阈值 T 的均值**——所以在计算器里，"90% 阈值"的平均上下文约为窗口的 46%，读税数字会低于本表。两者不必逐一对齐：本表用来讲直觉，计算器用来做决策（且计算器内部"省 X%"的对比始终自洽）。

### 从公式直接读出的规律

- **`T*` 与窗口 `W` 无关**（只要 `T* < W`）。所以最优**百分比**取决于窗口大小：
  同样参数下，1M 窗口 → 13%，200K 窗口 → ~65%。**窗口越大，最优压缩百分比越低。**
- `g` 越大、摘要 `S` 越大 → 阈值可调高（√ 关系，不敏感）；
- 基础价越贵（`h` 越大）→ 阈值该调低。

## 4. 为什么真实工具都在 ~75–95% 才压缩？

纯 token 成本只是问题的一半。三股力量把最优点大幅**右移**：

1. **信息损失（质量成本）**——压缩有损，变量名/报错细节/早期决策被压没，agent 变笨、返工。隐性成本通常远大于那点 token 钱。
2. **Token 放大回路（能直接推翻"早压缩"）**——压缩驱逐文件内容后，agent 又去**重读同样的文件**，实测把消耗放大 **10–20×**（Codex v0.118 因压缩过频导致 token 爆炸）。等于放大 `g` 和 `K`，最优点显著右移。
3. **TTL 过期**——5min 缓存只在**持续活跃**时有效。一旦空闲/长工具调用超时，缓存失效，下一轮要按 **1.25×（或 1h TTL 的 2×）重写整段上下文**，是读价的 12.5×。此场景下大上下文惩罚暴增，**更该保持小上下文**。

### 实测行为（2026）

- **Claude Code**：默认约 **95%**（VS Code 版约 **75%**，留 ~20% 给压缩本身），可用 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` 覆盖；社区经验 **40–70% 体验更好**。
- **Codex CLI**：阈值触发，硬上限 **90%**；本地路径先保留最近 ~20K token 再生成定长 handoff 摘要；OpenAI 托管模型走服务端加密快路径 `POST /v1/responses/compact`。

它们默认偏晚，主要在保**质量、防放大**，而非最小化 token 账单。

---

## 5. 结论与建议

1. **别用"总量/单次容量 = 压几次"算账**——忽略了每轮读税这个主导项。正确的决策变量是 `{g, S, 模型单价, TTL, 重读放大系数}`，用第 3 节的 √ 公式估。
2. **纯成本最优偏早（~10–25% 窗口）；计入质量 + 放大后大幅右移到 ~60–90%。** "40% vs 60%"之争在纯成本上都比"拖到 90%"省钱；真正要防的不是压几次，而是**别触发重读放大**。
3. **比"一刀切阈值压缩"更优的是结构化 context editing**——只清除陈旧 `tool_result`、保留摘要与近期消息，而非把整段历史揉成摘要。Anthropic 的 `clear_tool_uses` / context editing 正是干这个：同时压低读税、避免放大回路。**最推荐的方向。**
4. **盯 TTL**：活跃会话用默认 5min；慢节奏/会中断的会话考虑 1h TTL（写贵 2× 但存活久）或启动 pre-warm，别让缓存反复冷启动。

---

## 附录：模型假设与局限

- 假设每轮新增 `g` 近似恒定、摘要地板 `S` 近似固定、会话总轮数 `N` 与阈值无关。
- 未把"质量损失导致的返工轮数增加"货币化（这是把最优点右移的主因之一）。
- **重读放大**：现实中它会同时抬高每轮增长 `g`（重读的内容也进上下文）与每次压缩开销 `K_fixed`。配套计算器做了保守简化——只把 `amp` 乘到 `K_fixed`，因此对最优阈值 `T*` 的影响是 **√amp**（amp 从 1→20 约右移 4.5×）；若把 `g` 也一并放大，右移会更强。
- 读税倍率取 0.1×、写取 1.25×/2×（Anthropic 定价）；其他厂商倍率不同需替换。
