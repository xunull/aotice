# 重读放大 & 最优压缩阈值:公式逐项详解

> 这篇是给"看到公式里一堆字母就懵"的人写的。目标:把
> `T* = S + √(2·g·K_fix / h)` 里**每一个变量**讲成人话——它是什么、单位是什么、
> 典型值多少、数字从哪来。看完你应该能自己在纸上把最优阈值算出来。
>
> 配套:[成本建模总览](./上下文压缩时机的成本建模.md) · 计算器项目 `../compaction-calculator/`

---

## 0. 一分钟结论

- **要解决的问题**:上下文(对话历史)涨到多大时触发一次压缩(compaction),整体最省钱?
- **答案是一个公式**:`最优触发阈值 T* = 压缩地板 S + √(2·g·K_fix / h)`。
- **它在平衡两笔互相打架的钱**:上下文放着每轮要交"读税"(越大越贵,催你**早压**);但每压一次又有固定开销(催你**攒着晚压**)。平方根就是这两者的平衡点。
- **重读放大 `amp`** 是其中一个旋钮:压缩会把文件内容压没、agent 又去重读,这个浪费用 `amp` 量化,它会把最优点往后推。

---

## 1. 重读放大(amp)到底是什么

**一句话:压缩把文件内容压没了,agent 之后又得重新读一遍;这些"本来已经在上下文里、却被迫重读"的额外消耗,就是重读放大。**

### 为什么会发生

压缩是**有损**的。agent 读了 5 个文件(比如 40K token)才搞懂任务,压缩一触发,摘要只留下"读过 `auth.ts`、`db.ts`……"这种梗概,**文件的实际内容被丢掉了**。

下一步 agent 需要 `auth.ts` 里的某个细节 → 只能再调工具**把整个文件重新读进来**(8K token 又回到上下文)。再过一阵又压缩、又丢、又重读……形成循环。实测最惨的情况(Codex v0.118)能把真实 token 消耗放大 **10–20×**。

### 怎么给自己估一个值

看你的会话里,压缩之后 agent 有多爱重读文件:

| 你的场景 | amp 取值 |
|---|---|
| 几乎不重读(任务短、摘要够用) | 1 – 2 |
| 偶尔重读几个关键文件 | 3 – 5 |
| 老是把同几个大文件反复读回来 | 6 – 10+ |

`amp = 1` 表示"理想、无重读",是纯成本模型的下界。

## 2. 先看懂那张图:两笔在打架的成本

每一轮问答的成本里,**跟"压缩阈值"有关**的只有两笔,方向相反:

```
成本
 │＼                                   ╱  ← ② 压缩开销:阈值越低→压得越勤→越贵
 │  ＼                               ╱
 │    ＼_____                 _____╱
 │           ＼____     ____╱             ← 总成本 = ① + ②,U 形
 │                ＼_╳_╱   ← 谷底 = 最优阈值 T*
 │            ____╱     ＼____
 │       ____╱               ＼____
 │  ____╱                         ＼___   ← ① 读税:阈值越高→上下文越大→越贵
 └──────────────────────────────────────► 压缩触发阈值 T
        低                          高
```

- **① 读税(持有成本)**:上下文越大,每轮重读它就越贵 → 阈值定得越高越贵。**催你早压。**
- **② 压缩开销**:每压一次都有固定花费(生成摘要 + 重读放大),压得越勤(阈值越低)分摊到每轮就越贵。**催你晚压。**

两条曲线一升一降,加起来是 **U 形**,**谷底就是最省钱的阈值 `T*`**。公式就是在算这个谷底的横坐标。

## 3. 公式与每个变量(本文核心)

### 主公式

```
T*    = S + ΔC*                        ← 最优触发阈值 = 地板 + 最优增长量
ΔC*   = √( 2 · g · K_fix / h )         ← 两次压缩之间该让上下文涨多少
K_fix = amp · S · (P_out + P_write)    ← 每压一次的固定开销
h     = 0.1 · P_in                     ← 每 token 每轮的"读税"单价
```

### 变量总表(先扫一眼,下面逐个讲)

| 符号 | 名称 | 白话含义 | 单位 | Opus 4.8 示例 | 数字从哪来 |
|---|---|---|---|---|---|
| `T*` | 最优阈值 | 上下文涨到这么大就压缩,最省钱 | token | ~132K | **算出来的答案** |
| `S` | 压缩地板 | 压完缩回多大(摘要 + 保留的近期消息) | token | 20,000 | 你设定 / 工具决定 |
| `ΔC*` | 最优增长量 | 两次压缩之间让上下文涨多少 | token | ~112K | `= T* − S` |
| `g` | 每轮新增 | 每问答一轮往上下文加多少 token | token/轮 | 5,000 | 你的会话实测 |
| `h` | 持有单价(读税) | 每 token 每轮的"停车费" | $/(token·轮) | $0.5/M | `= 0.1 × P_in` |
| `K_fix` | 每次压缩开销 | 压一次要花多少钱(含重读放大) | $/次 | ~$0.63 | `= amp·S·(P_out+P_write)` |
| `amp` | 重读放大 | 压完 agent 重读文件的倍数 | 无量纲(≥1) | 1 | 你的会话实测 |
| `P_in` | 输入价 | 模型的输入 token 单价 | $/token | $5/M | models.dev |
| `P_out` | 输出价 | 生成摘要按这个计费 | $/token | $25/M | models.dev |
| `P_write` | 缓存写价 | 摘要写进缓存一次的单价 | $/token | $6.25/M(5min) | models.dev |

> 单位提示:`$/M` 就是"每一百万 token 多少美元"。换算成"每 token"要除以 1,000,000(即 `$5/M = 0.000005 $/token = 5e-6`)。手算时统一用"每 token"最不容易错。

### 逐个白话解释

**`T*` — 最优阈值(要求的答案)**
上下文(= system prompt + 全部历史对话)累计到 `T*` 这么多 token 时,就该触发一次压缩。低于它压得太勤(浪费在压缩开销),高于它上下文太肥(浪费在读税)。这是公式最终吐出来的数。

**`S` — 压缩地板**
压缩完成后,上下文不会变成 0,而是缩回到一个"地板":一段摘要 + 保留的最近几轮消息。这个大小基本固定(工具通常压到一个有界的目标尺寸,比如 Codex 保留最近 ~20K)。它是每次压缩的"起点"。

**`ΔC*` — 最优增长量(= T* − S)**
从地板 `S` 开始,上下文每轮涨一点,涨到 `T*` 再压回 `S`。`ΔC* = T* − S` 就是"每个压缩周期里,允许它长多高"。公式其实先算出 `ΔC*`,再加上 `S` 得到 `T*`。

**`g` — 每轮新增 token(需求率)**
agent 每完成一轮(你的消息 + 它的回复 + 工具返回的文件内容/命令输出),会往上下文里追加多少 token。工具调用多、读文件多的会话,`g` 就大(比如 5,000–10,000);纯聊天的会话 `g` 小。它决定上下文涨多快。

**`h` — 持有单价 / 读税(整个模型的关键)**
这是最反直觉、也最重要的一项。**开启 prompt caching 后,每一轮都要把整段已有上下文按"缓存读价"重新计费一遍**(因为每次请求都要把全部历史发给模型)。缓存读价 = 输入价的 **0.1 倍**。所以上下文里每有 1 个 token,你每一轮都要为它付 `h = 0.1 × P_in` 的"停车费"。上下文越大、轮数越多,这笔税二次增长——它才是长上下文真正烧钱的地方。

**`K_fix` — 每次压缩的固定开销**
压一次要花的钱,由三部分组成,再乘重读放大:
- 生成摘要:摘要有 `S` 个 token,按**输出价** `P_out` 计费 → `S · P_out`;
- 摘要首次写入缓存:按**缓存写价** `P_write`(5min TTL = 输入价的 1.25 倍)→ `S · P_write`;
- 乘以 `amp`:把压缩引发的重读浪费算进去。
合起来:`K_fix = amp · S · (P_out + P_write)`。

**`amp` — 重读放大**(见第 1 节)。注意它是**乘在 `K_fix` 上**的:压一次"顺带"要重读一堆文件,所以每次压缩的实际代价被放大。

**`P_in / P_out / P_write` — 三个价格**
分别是输入、输出、缓存写的单价,直接来自 [models.dev](https://models.dev/)(计算器用 `npm run sync:pricing` 自动同步)。缓存读价固定是 `0.1 × P_in`。

## 4. 公式为什么长这样(拿"进货"打比方)

这个公式不是我拍脑袋写的,它就是经济学里的**经济订货量(EOQ)**公式,原本用来算"仓库该多久进一次货"。对应关系:

| 进货问题 | 我们的问题 |
|---|---|
| 货放仓库里占资金(持有成本) | 上下文放着每轮交读税(`h`) |
| 每次下单有固定手续费 | 每次压缩有固定开销(`K_fix`) |
| 每天卖出多少货(需求率) | 每轮新增多少 token(`g`) |
| 最优每次进多少货 | 最优每周期让上下文涨多少(`ΔC*`) |

**看懂分子分母,就懂了整个公式:**

```
ΔC* = √( 2 · g · K_fix / h )
             └─分子─┘   └分母
```

- **分子 `g · K_fix`(在根号里往上顶)**:压一次越贵(`K_fix` 大)、上下文涨得越快(`g` 大)→ 就该**攒着多涨一点再压**(`ΔC*` 变大,少压几次来摊薄固定开销)。所以 `amp` 变大 → `K_fix` 变大 → 阈值往后推。
- **分母 `h`(往下压)**:读税越贵(`h` 大)→ 越不能让上下文变肥 → `ΔC*` 变小,**早点压**。
- **为什么开平方**:两笔成本一个随 `ΔC` 上升(读税)、一个随 `ΔC` 下降(压缩摊销),它们的平衡点数学上恰好落在平方根处。平方根还意味着**不敏感**:`amp` 翻 4 倍,阈值才移 2 倍——所以不用把参数估得特别准。

> **推导(可跳过)**:每轮阈值相关成本
> `c(T) = h·(S+T)/2 + [g/(T−S)]·K_fix`。
> (第一项:上下文从 `S` 涨到 `T`,平均大小是中点 `(S+T)/2`,乘读税;第二项:每周期 `(T−S)/g` 轮压一次,把 `K_fix` 摊到每轮。)
> 对 `T` 求导并令其为 0:`h/2 − g·K_fix/(T−S)² = 0`,解出 `(T−S)² = 2·g·K_fix/h`,开方即得 `ΔC*`。

## 5. 手把手算一遍(Opus 4.8,amp=1,S=20K,g=5000)

全部换成"每 token"单位(`$/M ÷ 1,000,000`):

**第 1 步 · 读税单价 `h`**
```
h = 0.1 × P_in = 0.1 × $5/M = $0.5/M = 0.0000005 $/token/轮  (即 5e-7)
```

**第 2 步 · 两个价格**
```
P_out   = $25/M   = 25e-6 $/token
P_write = 1.25 × P_in = 1.25 × $5/M = $6.25/M = 6.25e-6 $/token
```

**第 3 步 · 每次压缩开销 `K_fix`**
```
K_fix = amp · S · (P_out + P_write)
      = 1 × 20000 × (25e-6 + 6.25e-6)
      = 20000 × 31.25e-6
      = $0.625 / 次
```

**第 4 步 · 最优增长量 `ΔC*`**
```
ΔC* = √( 2 · g · K_fix / h )
    = √( 2 × 5000 × 0.625 / 5e-7 )
    = √( 6250 / 5e-7 )
    = √( 1.25e10 )
    ≈ 111,803 token
```

**第 5 步 · 最优阈值 `T*`**
```
T* = S + ΔC* = 20000 + 111,803 ≈ 131,803 ≈ 132K
   → 占 1M 窗口的 13.2%
```

**顺便验证成本(和计算器对得上)**
```
读税/轮     = h·(S+T*)/2 = 5e-7 × (20000+131803)/2 ≈ $0.0380
压缩摊销/轮 = g/(T*−S)·K_fix = 5000/111803 × 0.625 ≈ $0.0280
合计        ≈ $0.066/轮 → $6.6 / 100 轮  ✓
```

**结论**:最省钱是在 **~132K(13% 窗口)** 就压,远早于工具默认的 90%——因为读税在主导。

## 6. 三个你能自己验证的规律

1. **调大 `amp`(重读放大),谷底右移**:`amp` 1→20,`T*` 按 √20 ≈ 4.5 倍右移。这就是"压太勤会更贵、所以现实里晚压"的数学解释。
2. **换小窗口,最优百分比反而升高**:`T*` 的**绝对 token 数几乎不变**,所以 1M 窗口是 13%、200K 窗口就成了 ~65%。**窗口越大,最优压缩百分比越低。**
3. **`g` 或 `S` 变大,阈值升高但迟钝**(√ 关系);**输入价越贵(`h` 越大),阈值该调低**。

## 7. 局限(别把它当成唯一真理)

- 只算了**跟阈值有关的成本**:每轮固定的输出、增量写入等与阈值无关,是常数,不影响最优点,但也不在图里。
- 假设 `g` 恒定、`S` 固定;真实会话会波动。
- **没把"信息损失导致 agent 变笨、返工多花几轮"货币化**——这项会把最优点进一步右移,也是真实工具压在 75–95% 而非 13% 的主因。
- 计算器把 `amp` 做了保守简化:只乘到 `K_fix`(√amp 效应);若把 `g` 也一并放大,右移会更强。
