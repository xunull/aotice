# EOQ(经济订货量)与「闭式解」详解

> 我们求最优压缩阈值用的那个公式 `T* = S + √(2·g·K/h)`,不是拍脑袋写的——它是一个 1913 年就有的经典结论 **EOQ(经济订货量)** 的搬运。这篇把 EOQ 从零讲清楚:它解决什么问题、公式里的 √ 从哪来、"闭式解"是什么意思、以及怎么套到上下文压缩上。
>
> 关联:[成本建模总览](./上下文压缩时机的成本建模.md) · [公式详解](./重读放大与公式详解.md)

---

## 1. 先说「闭式解」是什么

**闭式解(closed-form solution)= 有一个现成公式,把输入代进去、用有限次标准运算(+ − × ÷ √)就能一步算出答案——不用试、不用搜、不用循环。**

对照另一种做法:

| | 闭式解(我们用的) | 数值解 / 暴力搜索 |
|---|---|---|
| 怎么求最优 | 代进公式,一次算完 | 把候选值从小到大试几百个,每个算一遍,挑最好的 |
| 速度 | 瞬间 | 慢,要扫很多点 |
| 精度 | 精确 | 取决于扫得多密 |
| 给不给直觉 | 给(看公式就知道每个变量往哪推) | 不给(只是一堆数) |

> 有意思的是,配套计算器里**两者都在用**:那条 U 形成本曲线是**数值**画的(取几百个阈值各算一次连成线),而那根标"最优"的**竖线是闭式解公式**一步定出来的——不是从曲线上慢慢找谷底。

---

## 2. EOQ 解决的经典问题:每次进多少货?

EOQ = **Economic Order Quantity(经济订货量)**,Ford W. Harris 于 1913 年提出,是库存管理最经典的公式。

**场景**:一家店卖货,货要向供应商订。两难:

- **一次订很多** → 库存堆积,占用资金、占仓库(**持有成本高**)。
- **一次订很少** → 很快卖光又得再订,每次下单都有固定手续费(**订货固定成本高**)。

到底一次订多少,总成本最低?这就是 EOQ 要回答的。

### 库存的"锯齿"图

进一批货,库存从 `Q` 开始,随着卖出线性下降到 0,再进一批,又回到 `Q`:

```
库存
 Q │＼        ＼        ＼
   │  ＼        ＼        ＼
   │    ＼        ＼        ＼          平均库存 = Q/2
 Q/2┄┄┄┄╲┄┄┄┄┄┄┄╲┄┄┄┄┄┄┄╲┄┄┄
   │        ＼        ＼        ＼
 0 └──────────＼────────＼────────＼──► 时间
      一个周期    下一批    再下一批
```

因为在 `Q` 和 `0` 之间线性来回,**平均库存 = Q/2**。这是后面推导的关键。

---

## 3. 推导:√ 是怎么冒出来的

定义三个量:

- `D` = 需求率(每单位时间卖出多少),单位 件/时
- `K` = 每次订货的**固定**成本(手续费),单位 $/次
- `h` = 每件货每单位时间的持有成本,单位 $/(件·时)
- `Q` = 每次订多少(我们要求的决策变量)

**两笔成本,方向相反:**

**① 订货成本**:每单位时间要订 `D/Q` 次(卖得快、每次订得少 → 订得勤),每次花 `K`:
```
订货成本 = K · D / Q     ← Q 越大越小(少订几次)
```

**② 持有成本**:平均库存 `Q/2`,每件每时 `h`:
```
持有成本 = h · Q / 2      ← Q 越大越大(囤得多)
```

**总成本**:
```
TC(Q) = K·D/Q + h·Q/2
```

一个随 Q 降、一个随 Q 升,加起来是 U 形,有个最低点。求最低点 = 对 Q 求导、令其为 0:

```
d TC/dQ = −K·D/Q² + h/2 = 0
        →  h/2 = K·D/Q²
        →  Q² = 2·K·D / h
        →  Q* = √( 2·K·D / h )      ★ 这就是 EOQ 公式
```

**√ 的来历**:因为总成本里有个 `1/Q` 项(订货)和一个 `Q` 项(持有),求导后变成 `1/Q²` 和常数,解出来 `Q²` = 某个东西 → 开平方。**只要"一笔成本 ∝ 1/批量、另一笔 ∝ 批量",最优批量必然是平方根形式**——这是 EOQ 的数学本质。

**平方根还意味着"迟钝"**:输入翻 4 倍,最优批量才翻 2 倍。所以参数估不准也没大关系。

## 4. 把 EOQ 套到上下文压缩

我们的问题长得和进货一模一样,只要换个说法:

| 进货问题 | 上下文压缩问题 |
|---|---|
| 货放仓库占资金(**持有成本 h**) | 上下文放着每轮交**读税**(缓存读价)`h` |
| 每次下单有**固定手续费 K** | 每次压缩有**固定开销** `K_fix` |
| 每天卖出多少(**需求率 D**) | 每轮新增多少 token(**增长率 g**) |
| 最优**每次订多少 Q** | 最优**两次压缩间涨多少 ΔC** |

直接把 `D→g`、`K→K_fix`、`Q→ΔC` 代进 EOQ:

```
ΔC* = √( 2 · g · K_fix / h )
T*  = S + ΔC*
```

### 一个小差别:我们的"锯齿"有地板

库存在 `Q` 和 **0** 之间来回,平均 `Q/2`。而上下文在阈值 `T` 和**地板 `S`** 之间来回(压缩压不到 0),平均是 `(S+T)/2`。展开:

```
平均上下文 = (S + T)/2 = S + ΔC/2
持有成本   = h·(S + ΔC/2) = h·S + h·ΔC/2
             └─常数─┘  └随 ΔC 变─┘
```

那个 `h·S` 是**常数**(不随 ΔC 变),对"找最低点"没影响,求导时直接消掉。剩下的 `h·ΔC/2` 部分和标准 EOQ 完全一样。**所以地板 `S` 不改变最优增长量 `ΔC*` 的公式,只是最后加回去得到 `T*`。**

---

## 5. 手算一遍(Opus 4.8)

单位统一成"每 token"($/M ÷ 1,000,000):

```
h     = 0.1 × 输入价 = 0.1 × $5/M = 5e-7   $/(token·轮)
K_fix = amp·S·(输出价 + 写价)
      = 1 × 20000 × (25e-6 + 6.25e-6) = $0.625 / 次
g     = 5000 token/轮

ΔC* = √( 2 · g · K_fix / h )
    = √( 2 × 5000 × 0.625 / 5e-7 )
    = √( 1.25e10 )
    ≈ 111,803 token

T*  = S + ΔC* = 20000 + 111,803 ≈ 132K   （1M 窗口的 13%）
```

**换成暴力搜索会怎样**:你得把 T 从 21K 试到 1M,每个值算一遍 `h·(S+T)/2 + g/(T−S)·K_fix`,画出曲线再找谷底——几百次计算。**闭式解一步到位,还告诉你规律**:`K_fix`↑(比如重读放大 amp 变大)→ 阈值↑;`h`↑(输入价贵)→ 阈值↓。

## 6. EOQ 的假设(也就是我们的假设)

经典 EOQ 成立需要几个理想化前提,套到压缩上就是我们模型的局限:

| EOQ 假设 | 对应到压缩 | 现实偏差 |
|---|---|---|
| 需求率 `D` 恒定 | 每轮增长 `g` 恒定 | 真实会话有波动 |
| 每次订货固定费 `K` 不变 | 每次压缩开销固定 | 摘要大小/重读会变 |
| 持有成本线性 | 读税按 token 线性 | 基本成立(缓存读线性计费) |
| 瞬时补货、不缺货 | 压缩瞬间完成、无质量损失 | **最大偏差**:压缩有损、会导致返工 |

最后一行是重点:经典 EOQ 不考虑"质量",而压缩的信息损失恰恰是现实里最贵的成本。**所以这个 EOQ 最优阈值是"纯 token 成本"的下界参照,不是最终答案**——真实决策要把质量因素叠上去,把最优点往后推(这也是为什么工具默认压在 75–95% 而非 13%)。

---

## 7. 一句话总结

**"EOQ 闭式解"= 这个"何时压缩"的问题,和百年前"每次进多少货"的库存问题数学同构,于是白捡了一个现成的平方根公式:代进参数就直接吐出最优压缩点,不用一个个阈值去试,还能从公式结构直接读出规律。** 代价是它继承了 EOQ 的理想化假设——尤其不含质量损失,所以给出的是成本下界,不是照着压的目标。
